# EDA PoC 可行性报告（基于 v1 需求）

来源文件：`docs/input/eda_requirements_v1.txt`（存在明显编码损坏，以下结论基于可识别术语与数字信号整理）。

## 1. 结论摘要
- 在“收敛范围后的 PoC”前提下，项目可行。
- 若按当前文本可见的全量目标一次性交付，风险偏高。
- 建议采用“两阶段路线”：先交付可验证 MVP，再扩展到多架构/更深 AI 优化。

### 明确标注的估算
- [Estimate] 缩范围 MVP 成功概率：`~80%`。
- [Estimate] 全量 v1 一次性交付成功概率：`~60%`。

## 2. 需求可行性分解
- 工艺与对象：聚焦 `3nm` CPU 场景，可扩展 `2nm`。
- 流程能力：`Verilog/VHDL -> DRC/LVS/ERC/STA/DFM -> GDSII/SPICE/报告` 的链路在工程上可落地。
- 生态兼容：与 `Synopsys/Cadence` 生态对接属于高价值但高依赖项。
- AI 增强：可先做“建议/排序/参数搜索”型能力，再逐步进入闭环优化。

## 3. 技术可行性
- 传统 EDA 能力（规则检查、时序分析、格式互通）成熟度高，可作为基线。
- 难点集中在：
  - 先进节点约束下的性能目标（时长与精度并存）。
  - AI 模型与 EDA 流程耦合后的可解释性与稳定性。
  - 数据与工具链授权边界（PDK/商业工具）。

### 明确标注的估算
- [Estimate] 关键指标中最难达成项：`72h` 全流程时长与高精度并行达标。
- [Estimate] 在限定设计规模下实现 `<=48h` 的 DRC/LVS/ERC 吞吐：技术上可达，但需较强算力与流程调优。
- [Estimate] AI 带来的稳定增益区间：`10%-30%`（取决于数据质量与任务定义）。

## 4. 资源可行性
- 运行环境信号与需求文档一致：
  - OS：`Windows 10/11 64-bit`、`Ubuntu 20.04 64-bit`
  - 语言/工具：`Java 11+`、`Python 3.8+`、`GCC/Clang/ModelSim/VCS`
- 建议计算资源以“可重复跑通大样例”为标准配置。

### 明确标注的估算
- [Estimate] 单套 PoC 执行节点建议：`>=16 vCPU, >=64GB RAM, >=2TB NVMe`。
- [Estimate] 若引入本地 AI 推理：`>=1 张 RTX 3090 级别 GPU`。

## 5. 进度可行性
- 建议拆分：
  - 阶段 A（MVP）：单架构 + 基础签核链路 + 基准 KPI。
  - 阶段 B（扩展）：多架构、更多 memory 场景、AI 深度优化。

### 明确标注的估算
- [Estimate] 阶段 A（MVP）周期：`10-14 周`。
- [Estimate] 阶段 B（扩展）周期：`12-24 周`。
- [Estimate] 由于原始需求文本损坏，前置澄清时间：`1-2 周`。

## 6. 主要风险与缓解
- 风险 1：需求语义不完整/歧义高。
  - 缓解：先完成“清洗版需求 + KPI 表 + 验收门槛”。
- 风险 2：外部依赖（PDK/商业工具许可）导致关键路径阻塞。
  - 缓解：建立依赖清单，按“可替代/不可替代”分级。
- 风险 3：AI 效果波动，难以稳定复现。
  - 缓解：固定数据集、固定评测脚本、保留传统基线回退路径。

### 明确标注的估算
- [Estimate] 当前阶段最高风险项：需求清晰度与外部许可依赖。
- [Estimate] 若不先做需求清洗，整体延期概率：`>40%`。

## 7. 建议的立项门槛（Go/No-Go）
- 必须项：
  - 清洗后的需求基线与验收指标签字。
  - 最小依赖（PDK/工具/算力）到位。
  - 基准设计与对照数据固定。
- 达成以上条件后，再进入开发执行更稳妥。
